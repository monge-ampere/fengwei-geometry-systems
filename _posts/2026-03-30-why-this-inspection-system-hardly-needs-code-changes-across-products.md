# Why Curved-Surface Pattern Inspection Systems Can Be Reused Across Products: Templates, Topology, and Tolerance in One XML File

> **English Abstract**  
> This article explains why the system can be reused across different products, regions, and production lines with minimal changes to the core DLL. The key lies not in writing universally adaptable code, but in organizing templates, topology, tolerance, block sequencing, and product entry points into an external XML-based geometric description layer.

如果只从外部看，很多人会以为这套系统之所以能在不同国别、不同产品甚至不同产线之间复用，主要靠的是“代码写得通用”。但真正做过这套系统的人知道，事情并不是这样。它真正的关键，不在于不断修改核心 DLL 去适配变化，而在于我当时把大量会变化的东西，提前组织进了一套外部几何模板文件里。对这套系统来说，xml 不是附属配置，而是模板、几何、拓扑、容错与调度逻辑的共同载体。也正因为如此，很多产品切换时，真正需要改动的并不是核心代码，而主要是图档、xml 和 ini 这一层。

为了避免一上来就陷入字段细节，先看这份 xml 的整体结构。只有先看到它的层次，后面关于模板、几何锚点、block 和产品入口的讨论才不会显得零散。

```xml
<entity>
    <!-- 字符 / 图样结构模板库 -->
    <character>
        <skeleton name="Japan" size="149" genus="1">
            <p x="..." y="..." />
            <p x="..." y="..." />
            <p x="..." y="..." />
            <!-- ... -->
        </skeleton>

        <skeleton name="G" size="58" genus="1">
            <p x="..." y="..." />
            <p x="..." y="..." />
            <!-- ... -->
        </skeleton>

        <skeleton name="R" size="57" genus="1">
            <p x="..." y="..." />
            <!-- ... -->
        </skeleton>

        <skeleton name="P" size="48" genus="1">
            <p x="..." y="..." />
            <!-- ... -->
        </skeleton>

        <!-- 其余出现在项目中的字符 / 图样 block 也都在这里定义 -->
        <!-- ... -->
    </character>

    <!-- 全局几何锚点 -->
    <globalGeo size="...">
        <Geo name="Apple" size="5" extValue="0" smode="0" type="1">
            <p x="..." y="..." prio="1" />
            <p x="..." y="..." prio="1" />
            <!-- ... -->
        </Geo>

        <Geo name="Japan" size="4" extValue="0" smode="0" type="1">
            <p x="..." y="..." prio="1" />
            <!-- ... -->
        </Geo>

        <Geo name="sn" size="12" extValue="1111" smode="0" type="2">
            <p x="..." y="..." prio="1" />
            <p x="..." y="..." prio="1" />
            <!-- ... -->
        </Geo>

        <!-- 其余全局几何锚点 -->
        <!-- ... -->
    </globalGeo>

    <!-- 当前产品的 block 序列 -->
    <blockSet size="...">
        <block prio="1" extValue="0" type="1" />
        <block prio="1" extValue="0" type="1" />
        <block prio="1" extValue="0" type="2" />
        <!-- ... -->
    </blockSet>

    <!-- 局部擦除 / 屏蔽规则 -->
    <eraseSet size="1">
        <erase dock0="right" dock1="begin" enterDist="0" range="0" type="1" />
    </eraseSet>

    <!-- 产品模板图入口 -->
    <templPathSet size="2">
        <product templatePath="RR1.bmp" rows="..." cols="..." xOffset="..." yOffset="..." type="1" angle="..." />
        <product templatePath="RR2.bmp" rows="..." cols="..." xOffset="..." yOffset="..." type="2" angle="..." />
    </templPathSet>
</entity>
```
上面这段代码只保留了 xml 的主干结构。实际文件中，`character` 下会列出项目里出现的各类字符或局部图样 block 的结构模板，`globalGeo` 下则保存对应的全局几何锚点。这里省略了大量具体点坐标，只保留其组织方式，是为了更直观地展示这份模板文件背后的几何逻辑。

如果先不陷入字段细节，而只抓住这份 xml 的主干结构，那么它做的事情其实非常明确：一方面保存字符或局部图样的结构模板，另一方面保存全局几何锚点、block 组织方式、局部擦除规则以及产品模板入口。也就是说，它并不是单纯存一张参考图，而是在系统层面描述：这个产品上有哪些可检测对象，它们大致位于哪里，哪些局部结构要被拿来做注册与测量，哪些区域需要额外容错，以及不同产品面分别对应哪一套模板入口。

从这个角度看，这份 xml 更接近一份**产品几何描述文件**，而不是普通意义上的配置文件。其内部层次也很清楚：`character` 下组织的是多个 `skeleton`，也就是字符或图样的局部结构模板；`globalGeo` 下保存的是命名的全局几何点集，用来提供更上层的定位锚点；`blockSet` 定义了当前产品实际参与检测的 block 序列；`eraseSet` 给出局部擦除与屏蔽规则；`templPathSet` 则负责把具体产品模板图接入系统。正是这些层次叠加在一起，这份 xml 才不只是“存模板”，而是在描述系统如何理解一个产品。

## 一、character：这里存的不是“字”，而是字符/图样的结构模板

很多人第一次看到 xml 里的 `character`，会下意识把它理解成“文本内容”。其实不是。这里真正保存的，并不是字符语义，而是字符或局部图样的**结构表达**。在精简版里，像 `Japan`、`G`、`R`、`P`、`A`、`B`、`C`、`D`、`E` 这些对象，下面都挂着一组 `p(x,y)` 点，它们共同构成了该字符或图样的局部模板。最常见的情况下，这类点集就是 medial axis，也就是中轴/骨架点集；更一般地说，它们是一类可用于注册与测量的特征点模板。

这件事很重要。因为一旦模板保存的不是图片，而是点集结构，后面的系统行为就会完全不同。图片只能拿来“比”，而点集结构既可以用来配准，也可以用来测量，还可以继续挂接拓扑信息、点类型和法向等更深一层的几何语义。也就是说，从这一层开始，这套系统就已经不是在做简单模板匹配，而是在做结构模板驱动的检测。

## 二、genus：模板不仅有几何，还有拓扑

这份 xml 里每个 `skeleton` 都带了 `genus`。如果不解释，这个字段会显得很怪；但它其实非常有意思。这里的 `genus` 指的是亏格，也就是对象从骨架角度看具有多少环柄或孔洞结构。换句话说，这套系统不是把字符模板只看成一堆点，而是进一步把它们看成带有**拓扑类别**的结构对象。

这一点很值得注意。它说明这套系统并不是在“尽量把工程写简单”，而是在真正有价值的地方引入了很少但很有效的数学信息。一个字符或者图样，仅靠局部几何位置，有时会因为形变、缺损、提取误差而变得不够稳；但一旦把亏格也带进来，模板匹配和结构选择就多了一层更稳的先验。它未必总是决定性的，但在复杂现场里，往往足以提高成功率。

## 三、globalGeo：这不是细节模板，而是全局几何锚点

如果说 `character/skeleton` 负责局部结构模板，那么 `globalGeo` 负责的就是更上层的几何定位。精简版里可以看到很多命名的 `Geo`，比如 `Apple`、`Japan`、`A1602`、`sn` 等，它们各自保存了一组点。尤其 `sn` 这一项最有意思，它不是一张图片，而是一组离散的槽位点，恰好对应动态字符串可能出现的位置。

这说明 `globalGeo` 不是给人看的“辅助信息”，而是给系统做**粗定位和几何组织**用的锚点层。它一方面帮助系统知道某个局部结构大致在哪里，另一方面也把动态内容——例如 SN code——纳入了同一几何框架里。于是系统处理的就不再只是“静态模板”，而是“静态结构 + 动态业务输入”的统一检测对象。

## 四、blockSet：系统真正工作的，不是整张图，而是一组 block

这份 xml 里还有一层特别关键，就是 `blockSet`。它定义了当前产品上到底有哪些 block，也就是有哪些基本检测单元。精简版里，`blockSet size="25"`，同时又有两个产品模板入口 `type="1"` 和 `type="2"`。这并不是巧合，而是在说明：这套系统在产品层已经把整个 pattern 切成了若干个基本工作单元，并按模板类别组织好了。

这也解释了为什么系统能并行，为什么不同区域可以分别定位、分别测量、分别输出结果。因为系统真正处理的，不是“整张图像里有什么”，而是“这 25 个 block 中，每一个是否在自己的几何位置上以正确方式出现了”。一旦这样组织，后面的两次配准和局部测量才有落脚点。

## 五、extValue：容错不是模糊地放大，而是方向化编码

这份 xml 里还有一个很值得注意的设计，就是 `extValue`。它并不是简单地给 block 增加一个统一 margin，而是把扩展量编码成了四个方向上的显式规则：左、右、上、下分别由不同位上的数字决定，再统一乘以扩展倍数。这样做的意义很直接：前级提取出来的 block 在现场往往存在误差，而这种误差通常又不是均匀的，不同方向上的偏移风险并不相同。把容错写成方向化的几何规则之后，系统就不必把这类补偿零散地留给后续代码去处理，而是可以在模板描述层面提前吸收一部分定位误差。

这就是我说这份 xml 很有“系统之美”的原因。它不是靠一个神秘参数去赌运气，而是把容错做成了**几何上可解释、工程上可维护**的规则。这样一来，成功率的提升不是来自运气，而是来自设计。

## 六、eraseSet 与 templPathSet：模板不是死的，产品也不是单入口

`eraseSet` 和 `templPathSet` 这两层，看起来像工程尾巴，其实很能体现这套系统的成熟度。前者说明模板不是死存储，而是允许做局部擦除和结构性屏蔽；后者说明系统并不是只面对单一模板图，而是可以根据产品类型切到不同模板入口，同时保留各自的尺寸、偏移和角度信息。

这两件事加在一起，就使得 xml 不再只是“模板文件”，而更接近一份**产品几何描述文件**：它既告诉系统“你要看什么”，也告诉系统“你不用看什么”；既告诉系统“模板图从哪里进来”，也告诉系统“局部结构该如何组织和扩展”。

## 七、所以，这份 XML 真正描述的不是配置，而是“系统如何看待产品”

如果一定要用一句话概括这份 xml，我不会说它是“检测配置文件”，也不会说它只是“模板文件”。更准确的说法是：

> **它是一份把字符库、局部结构模板、全局几何锚点、block 组织、拓扑先验、方向容错和产品模板入口统一到一起的几何描述文件。**

这句话听起来有点长，但它比“配置文件”接近真实得多。因为真正支撑这套系统跨国别、跨产品、跨产线复用的，恰恰不是不断改代码，而是这种把变化压缩进模板描述层的能力。今天回头看，我仍然觉得这份 xml 很能代表这套系统的气质：它不浮夸，但很有思想；它不只是保存模板，而是在保存系统对产品的理解方式。
